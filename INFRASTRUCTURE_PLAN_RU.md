## Обзор инфраструктуры и дорожная карта улучшений

Этот проект уже поставляется с ориентированным на продакшен стеком деплоя, основанным на:

- **GitHub Actions** для CI/CD с переиспользуемыми workflow.
- **Оркестрации Docker на одном сервере** через `docker-compose.local.yml` (приложение, nginx, MongoDB, Redis, опциональный стек метрик с Prometheus + Grafana + Loki и т.д.).
- **Blue/green‑деплоях**, реализованных через `scripts/local-containers-run.sh` и логику GitHub Actions.
- **Автоматическом управлении TLS‑сертификатами** с помощью контейнеров Certbot и общих Docker‑volume.

Для многих маленьких и средних проектов эта схема — сильная отправная точка: она даёт воспроизводимые деплои, health‑checks, наблюдаемость и безопасный способ выкатывать новые версии без мгновенного падения продакшена.

Дальше инфраструктуру логично развивать в двух направлениях:

- **План A – «Укреплённый стек на одном сервере»**: сохранить текущую архитектуру (один VPS + Docker Compose), но сделать инфраструктуру воспроизводимой и более безопасной с помощью Infrastructure as Code (Terraform), лучшего управления секретами и доступа, а также понятных операционных регламентов.
- **План B – «Кластерный стек на Kubernetes»**: постепенно перейти к деплою на многонодовый кластер Kubernetes с полноценным ingress‑контроллером (Traefik или nginx), управляемыми БД и GitOps‑подходом.

Эти планы не взаимоисключающие:

- План A — это **малозатратный путь с высокой отдачей**, если нужно повысить надёжность и повторяемость без резкого усложнения.
- План B — **долгосрочная опция**, когда один сервер перестаёт устраивать по масштабируемости, отказоустойчивости или требованиям компании.

Этот документ задуман как:

- **Дорожная карта улучшения инфраструктуры** для людей.
- **Высокоуровневая спецификация** для AI‑агентов, помогающих готовить и реализовывать тот или иной вариант деплоя (План A или План B).

## План A – Укреплённый стек на одном сервере (Terraform + улучшенные операции)

### Цель

Сохранить текущую архитектуру «один сервер + Docker Compose», но сделать её:

- **Воспроизводимой**: сервера и DNS можно поднимать из кода.
- **Более безопасной**: лучше управлять секретами и доступом по SSH.
- **Удобной в эксплуатации**: описать и заскриптовать типовые операции.

План A предполагает, что вас в целом устраивает:

- Один (или очень небольшое число) VPS‑инстансов.
- Docker Compose как основной уровень оркестрации.
- Текущая логика blue/green‑деплоя в скриптах и workflow.

### Ключевые изменения

1. **Внедрить Terraform для инфраструктуры**
   - Описать в коде:
     - VPS‑инстансы (compute) для `stage` и `prod`.
     - Security‑groups / firewall‑правила (SSH 22, HTTP 80, HTTPS 443).
     - DNS‑записи для `nextjs-super-boilerplate.visn-ai.io` (и других доменов) у выбранного DNS‑провайдера.
   - Хранить Terraform‑state удалённо (например, S3/GCS + блокировка стейта).
   - Использовать отдельные workspace или state‑файлы для `stage` и `prod`.

2. **Укрепить доступ к серверу**
   - Заменить аутентификацию по **SSH‑паролю** на:
     - SSH‑ключи, хранимые в надёжном месте (например, защищённый паролем ключ в секрет‑менеджере).
     - Использование SSH‑ключей GitHub Actions вместо `server_password` в секретах workflow.
   - По возможности:
     - Ограничить вход по SSH через `AllowUsers`.
     - Отключить вход под root.
     - Ограничить доступ по IP.

3. **Явно формализовать окружения деплоя**
   - Сохранить текущие workflow GitHub Actions, но:
     - Жёстко разделить инфраструктуру и стейт для `stage` и `prod` (переменные и workspace в Terraform).
     - Добиться единообразного формирования `.env.stage` и `.env.prod` (из GitHub Secrets) и описать это в документации.
   - Для AI‑агентов: рассматривать каждое окружение как отдельную цель со своими Terraform‑переменными и секретами.

4. **Улучшить управление секретами и конфигами**
   - Продолжать использовать GitHub Secrets как источник правды, но:
     - Ввести единый нейминг (`WEB_ENV_STAGE`, `WEB_ENV_PROD` и т.д.).
     - При необходимости добавить SOPS или облачный secret‑manager (AWS/GCP/Azure) для ротации ключей и централизованного доступа.
   - Документировать, как формируются `.env.*` файлы и где именно живут креды БД.

5. **Операционные плейбуки**
   - Описать и автоматизировать типичные операции:
     - Очистка диска (`./scripts/local-containers-run.sh clean` и `prune-images`).
     - Ручное продление сертификатов (`renew-certs`).
     - Сброс volume MongoDB при повреждённых/ошибочных данных.
   - Для AI‑агентов: использовать эти скрипты как **первую линию инструментария**, а не разовые команды docker.

6. **Стратегия метрик и мониторинга**
   - На уровне окружений принять решение, где включать стек метрик:
     - Для маленьких VPS (1 vCPU) держать `metrics_enabled: false` в проде и включать метрики только временно для диагностики.
     - Для серверов с 2+ vCPU — держать метрики включёнными постоянно.
   - Конфигурации Grafana и Prometheus хранить в этом репозитории под версионным контролем.

### Результат

После внедрения Плана A:

- Инфраструктуру (серверы, DNS, firewall) можно **воспроизвести с нуля** с помощью Terraform.
- Поток деплоя (GitHub Actions → сервер → Docker Compose) почти не меняется, но становится **более предсказуемым**.
- Операционные знания кристаллизуются в скриптах и документации, что упрощает жизнь и людям, и AI‑агентам.

## План B – Кластерный стек на Kubernetes (K8s + Ingress + GitOps)

### Цель

Эволюционировать от деплоя на одном сервере с Docker Compose к **архитектуре на базе Kubernetes**, которая обеспечивает:

- Несколько нод и более высокую отказоустойчивость.
- Нативные механизмы rolling‑обновлений, canary‑релизов и масштабирования.
- Более чёткое разделение ответственности между инфраструктурой, платформой и приложением.

План B **существенно сложнее** и обычно имеет смысл после того, как большая часть Плана A уже реализована (Terraform, секреты, базовые операционные практики).

### Ключевые изменения

1. **Определить платформу Kubernetes**
   - Выбрать вариант Kubernetes:
     - Управляемый кластер (EKS, GKE, AKS и т.п.).
     - Или self‑managed кластер поверх VM / bare metal (только если есть сильная команда ops/SRE).
   - С помощью Terraform описать:
     - Сам кластер.
     - Сетевую инфраструктуру (VPC/VNet, подсети, security‑группы, балансировщики).
     - Зону и записи DNS для доменов.

2. **Добавить Ingress‑контроллер (Traefik или nginx‑ingress)**
   - Заменить текущую связку nginx + Certbot на:
     - Ingress‑контроллер Kubernetes (Traefik — отличный кандидат, nginx‑ingress тоже подойдёт).
     - **cert-manager** для управления сертификатами Let’s Encrypt через Kubernetes‑ресурсы.
   - Настроить:
     - `Ingress`‑ресурсы, маппящие `nextjs-super-boilerplate.visn-ai.io` на сервис приложения.
     - HTTP → HTTPS редиректы, TLS и, при необходимости, роутинг по путям для нескольких приложений.

3. **Упаковать приложение под Kubernetes**
   - Подготовить манифесты или Helm‑чарт для:
     - `Deployment` + `Service` для Docker‑образа Next.js приложения.
     - Опционально `StatefulSet` или внешние управляемые сервисы для MongoDB/Redis (предпочтительно использовать managed‑БД).
   - Перенести идеи из Docker Compose в эквиваленты K8s:
     - Лимиты/резервы ресурсов → `resources` в Pod‑спеке.
     - Health‑checks → `livenessProbe` / `readinessProbe`.
   - Для blue/green / rolling‑обновлений:
     - Использовать стратегию Deployment (RollingUpdate) или отдельный инструмент (Argo Rollouts, Flagger) для канареек/blue‑green.

4. **Наблюдаемость в Kubernetes**
   - Заменить compose‑стек метрик на:
     - Prometheus (или Prometheus Operator).
     - Grafana.
     - Loki + Promtail или другую систему логирования.
   - Публиковать дашборды через Ingress, аналогично текущему подходу с nginx.

5. **Адаптация CI/CD**
   - Сохранить GitHub Actions как CI, но поменять шаги деплоя:
     - Сборка и пуш Docker‑образа (GHCR или другой реестр).
     - Применение манифестов через `kubectl` или переход к GitOps (Argo CD / Flux), когда конфиг‑репозиторий сам синхронизируется с кластером.
   - Для AI‑агентов:
     - Считать деплой изменением Kubernetes‑манифестов или значений Helm, а не `scp` + удалённый Docker Compose.

6. **Стратегия миграции с Compose на Kubernetes**
   - Начать со **stage‑кластера**:
     - Развернуть приложение и поддерживающие сервисы параллельно существующему stage на Docker Compose.
     - Прогнать smoke‑тесты, TLS и проверки масштабирования.
   - Постепенный перенос прод‑трафика:
     - Либо переключить DNS на новый Ingress.
     - Либо использовать слой распределения трафика / canary‑механику для плавной миграции.

### Результат

После внедрения Плана B:

- Приложение работает на **кластерной**, более отказоустойчивой платформе.
- Деплой описывается через **ресурсы Kubernetes** (YAML/Helm), что упрощает:
  - Горизонтальное масштабирование.
  - Интеграцию с service‑mesh, системами политик и продвинутой безопасностью.
- Связка ingress + cert‑manager заменяет кастомные скрипты вокруг nginx + Certbot и сокращает объём самописной операционной логики.

  - Спроецировать текущие сервисы из Docker Compose на ресурсы Kubernetes.
