## Дорожная карта улучшений инфраструктуры (детальный план действий)

Этот файл описывает **алгоритм наших действий** по каждому из намеченных пунктов улучшения текущего стека.  
Его цель — не потерять контекст между сессиями и дать людям и AI‑агентам понятный, пошаговый план.

## 1. Рейтлимиты и ответы 429 (защита от спама / pseudo‑DDoS)

### 1.1. Исследование текущих точек входа

1. Проанализировать, какие именно точки принимают внешний трафик:
   - Next.js API‑роуты и страницы (SSR/ISR/SPA).
   - Nginx‑конфигурацию (entry‑points: 80/443).
2. Составить карту:
   - “публичные” роуты (например, auth, формы, API‑эндпоинты).
   - особо чувствительные к спаму (логин/регистрация, отправка писем, webhooks).

### 1.2. Выбор уровня реализации рейтлимитов

3. Сравнить варианты:
   - На уровне **nginx** (например, `limit_req`, `limit_conn`).
   - На уровне **приложения** (middleware в Next.js, хранение счётчиков в Redis/Mongo).
   - На уровне **внешнего периметра** (Cloudflare, reverse‑proxy, WAF) — если используется.
4. Принять решение по архитектуре:
   - Базовая защита на уровне nginx (простая защита от очевидного спама/сканов).
   - Более тонкая бизнес‑логика рейтлимитов в коде (по пользователю/токену/ключу API).

### 1.3. Проектирование правил рейтлимита

5. Определить метрику:
   - запросы в минуту / секунду,
   - окно (fixed window / sliding window),
   - по какому ключу (IP, userId, API key, комбинации).
6. Определить целевое поведение:
   - возращать `429 Too Many Requests` с телом/заголовками `Retry-After`.
   - логировать события превышения лимита отдельно (для анализа атак).
7. Заложить “белые списки”:
   - внутренние IP / health‑check’и.
   - системные сервисы (если есть).

### 1.4. Реализация PoC

8. Реализовать минимальный PoC:
   - Nginx‑уровень (простое ограничение по IP на самые чувствительные роуты).
   - Либо middleware в Next.js API‑роутах с хранением счётчиков в Redis (если Redis включаем).
9. Добавить базовые метрики/логи:
   - количество 429 по роутам, IP, userId.

### 1.5. Нагрузочное тестирование и тюнинг

10. Подготовить нагрузочные тесты (например, `k6`, `autocannon`, `wrk`):
    - сценарии “нормального” трафика;
    - сценарии всплесков и “злого” трафика.
11. Прогнать тесты и оттюнить:
    - лимиты,
    - интервалы,
    - исключения / белые списки.

### 1.6. Продуктивный rollout и документация

12. Включить рейтлимиты сначала на stage, затем на prod.
13. Описать:
    - где реализован рейтлимит (nginx / app),
    - какие лимиты действуют,
    - где смотреть логи/метрики,
    - как временно ослабить/отключить при инцидентах.

## 2. Анализ генерации роутов и оптимизация рендеринга Next.js

### 2.1. Инвентаризация типов роутинга

1. Составить список страниц/роутов:
   - `SSR` (рендер на каждый запрос),
   - `SSG` (статические),

2. Для каждого критичного роута ответить на вопрос:
   - “рендерим ли мы на каждый запрос?”
   - “можно ли перейти в SSG/ISR или более агрессивный кэш?”

### 2.2. Анализ поведения в проде

3. Посмотреть:
   - как сейчас настроены заголовки кэширования (через nginx и Next.js).
   - нет ли роутов, которые зря рендерятся SSR без необходимости.
4. При необходимости включить дополнительное логирование:
   - время рендера,
   - частоту обращений к конкретным роутам.

### 2.3. Проектирование оптимизаций

5. Для каждого проблемного роута решить:
   - перевести ли его в SSG / ISR;
   - добавить ли `revalidate` (ISR) вместо полного SSR;
   - изменить логику кэширования на стороне nginx (например, cache‑zone).
6. Согласовать приоритеты:
   - сначала — самые тяжёлые и часто вызываемые страницы.

### 2.4. Реализация и тестирование

7. Поочерёдно вносить изменения:
   - настройки Next.js (тип страницы, revalidate).
   - при необходимости — конфиги nginx (кэш статики/HTML).
8. Проверять:
   - TTFB и общее время ответа;
   - корректность данных (нет ли устаревшего контента при агрессивном кэше).

### 2.5. Документация по стратегиям рендеринга

9. Добавить в документацию:

## 4. Документация по подключению к MongoDB в контейнере на сервере

### 4.1. Сбор технических деталей

1. Зафиксировать:
   - имя Mongo‑сервиса в `docker-compose.local.yml`;
   - используемую сеть (`service-api-network`);
   - порты (expose/ports).
2. Уточнить:
   - значения `MONGO_USER`, `MONGO_PASSWORD`, `MONGO_DB`;
   - как они попадают в контейнер (env из `.env.*`).

### 4.2. Описание способов подключения

3. Описать **локальное подключение с сервера**:
   - `mongo mongodb://MONGO_USER:MONGO_PASSWORD@mongo:27017/MONGO_DB` внутри сети Docker.
4. Описать **удалённое подключение с dev‑машины** (порт Mongo наружу не пробрасывается):
   - подключение через SSH‑туннель до IP контейнера Mongo:
     1. На сервере получить IP контейнера Mongo в сети `service-api-network`:

        ```bash
        docker inspect mongo | jq -r '.[0].NetworkSettings.Networks["service-api-network"].IPAddress'
        # пример: 172.18.0.2
        ```

     2. На dev‑машине открыть SSH‑туннель:

        ```bash
        ssh -L 27017:172.18.0.2:27017 user@server
        ```

        где `172.18.0.2` — IP контейнера из шага 1.

     3. Подключаться к Mongo локально по адресу:

        ```text
        mongodb://MONGO_USER:MONGO_PASSWORD@localhost:27017/MONGO_DB?authSource=admin
        ```
5. Если порт пробрасывается наружу (по необходимости) — явно описать риски безопасности и рекомендованный вариант (предпочтительно через туннель).

### 4.3. Примеры для инструментов

6. Дать примеры строк подключения:
   - для CLI (`mongosh`):
     - локальная Mongo в docker‑compose на сервере:
       - `mongosh "mongodb://MONGO_USER:MONGO_PASSWORD@mongo:27017/MONGO_DB"`
     - через SSH‑туннель (как описано выше, через IP контейнера):
       - `mongosh "mongodb://MONGO_USER:MONGO_PASSWORD@localhost:27017/MONGO_DB?authSource=admin"`
   - для GUI‑клиентов (MongoDB Compass, VS Code plugins):
     - **шаги для Compass**:
       1. Открыть MongoDB Compass → `New Connection`.
       2. Ввести строку подключения:
          - при работе через SSH‑туннель:
            - `mongodb://MONGO_USER:MONGO_PASSWORD@localhost:27017/MONGO_DB?authSource=admin`
          - при локальном docker‑compose (dev‑машина):
            - если порт проброшен как `27017:27017` → `mongodb://MONGO_USER:MONGO_PASSWORD@localhost:27017/MONGO_DB?authSource=admin`.
       3. Нажать `Connect` и убедиться, что видна база `MONGO_DB`.
7. Добавить раздел “частые ошибки”:
   - неверный пользователь/пароль,
   - неправильный хост (mongo vs localhost),
   - firewall / закрытый порт.

### 4.4. Встраивание в основной README / docs

8. Добавить ссылку на эту документацию:
   - из основного `README` (раздел про Mongo),
   - из FAQ.

## 5. Бэкапы MongoDB (локальная база в контейнере)

### 5.1. Определение требований к бэкапам

1. Определить:
   - RPO (максимально допустимая потеря данных),
   - RTO (время восстановления),
   - желаемую частоту бэкапов (ежедневно, ежечасно и т.п.).
2. Оценить объём БД и темпы роста:
   - чтобы понимать требования к диску и времени бэкапа.

### 5.2. Выбор инструмента и формата бэкапа

3. Основной вариант:
   - `mongodump` / `mongorestore` внутри контейнера или отдельного служебного контейнера.
4. Альтернативы:
   - файловые снапшоты volume (при поддержке провайдера/файловой системы),
   - managed‑бэкапы, если Mongo переедет в управляемый сервис.

### 5.3. Проектирование и реализация скриптов бэкапа

5. Спроектировать:
   - отдельный скрипт/служебный контейнер для бэкапа,
   - место хранения дампов (отдельный volume + копирование в object storage: S3/GCS/другой).
6. Реализовать:
   - скрипт `backup-mongo.sh` (условное имя) с параметрами:
     - имя БД,
     - путь/бакет для хранения,
     - теги бэкапа (дата/время, окружение).
7. Настроить расписание:
   - cron/systemd timer на сервере,
   - либо отдельный cron‑контейнер в Docker.

### 5.4. Алгоритм восстановления и отката

8. Разработать пошаговый сценарий восстановления:
   - остановка приложения/запись‑трафика в Mongo (при необходимости),
   - очистка/резервное копирование текущих данных,
   - `mongorestore` из нужного дампа,
   - проверка целостности и запуск приложения.
9. Учитывать характеристики сервера:
   - **1 vCPU**:
     - запуск бэкапа/восстановления в “щадящем” режиме,
     - возможная остановка части сервисов на время восстановления, чтобы не забивать I/O.
   - **несколько vCPU**:
     - можно позволить более агрессивный параллелизм (опции `--numParallelCollections` и т.п.);
     - меньше риск “забить” всё ядро и положить приложение.

### 5.5. Тесты восстановления

10. Регулярно (например, раз в месяц/квартал) проводить **тестовое восстановление**:
    - на отдельном стенде/сервере;
    - по реальному бэкапу из продакшена (с обезличиванием, если нужно).
11. Зафиксировать:
    - время, необходимое для полного восстановления;
    - возможные проблемные места (узкие места по CPU/диску).

### 5.6. Документация и интеграция с мониторингом

12. Задокументировать:
    - где хранятся бэкапы,
    - как запустить ручной бэкап,
    - как восстановиться на конкретную дату.
13. Добавить мониторинг:
    - алерты, если бэкап не прошёл/не был создан,
    - базовую статистику по размеру бэкапов и времени выполнения.
